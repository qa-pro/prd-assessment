# PRD Design Checklist

> 面向设计师评审 PRD：确认业务目标、体验范围、资产依赖是否足以支持设计方案产出，避免在 PRD 阶段就锁定具体视觉稿。

## 业务与用户洞察（P0）
- [ ] PRD 说明该版本的目标用户、核心场景、痛点及诉求数据来源。
- [ ] 成功指标、体验目标（满意度、转化、留存等）明确，可指导设计评估。
- [ ] 业务限制或假设（预算、时间、地域、运营策略）对体验的约束被列出。

## 体验范围与流程（P0）
- [ ] PRD 列出涉及的页面/模块/入口/端到端流程，并标注新增、改造或复用关系。
- [ ] 关键任务流程、状态流、场景地图在 PRD 中可追溯，便于设计梳理信息架构。
- [ ] 跨角色、跨终端、跨系统交互的边界与责任说明清楚。

## 交互特性（P0）
- [ ] 特殊交互（拖拽、批量、实时协作、快捷键、可视编辑等）在 PRD 中说明动线与约束。
- [ ] 状态、空态、加载、错误、权限受限等场景的业务期望被列举。
- [ ] 动效、反馈、提示（提示条、Toast、引导等）的业务要求被写明。

## 信息架构与内容（P1）
- [ ] 内容层级、导航、信息密度的业务诉求写在 PRD，帮助确定 IA。
- [ ] 文案/术语要求、语调规范、合规限制可在 PRD 中追溯。
- [ ] 多语言、多货币、单位/格式等本地化要求明确。
- [ ] 数据呈现（图表、报表、指标卡等）的业务含义与刷新频率说明清楚。

## 多终端/环境（P1）
- [ ] 必须支持的终端（Web、H5、小程序、App、客户端等）与适配策略写入 PRD。
