## 1. 评分框架总览

本标准采用**五维综合评分机制**，总分100分，从产品、后端、前端、设计、测试五个专业视角全方位评估 PRD 质量。

| 维度 | 权重 | 核心考核点 |
| :--- | :--- | :--- |
| **产品视角** | **40%** | 需求完整度、可执行性、一致性、体验交互、用户价值 |
| **后端视角** | **15%** | 业务范围、技术方案准备度、系统依赖、测试验收、性能容量 |
| **前端视角** | **15%** | 交互范围、设计交付、数据接口、状态异常、性能体验 |
| **设计视角** | **15%** | 业务洞察、体验范围、交互特性 |
| **测试视角** | **15%** | 场景覆盖、判定口径、状态异常、性能容量 |

## 2. 评分细则

### 2.1 产品视角 (40分)

#### 2.1.1 需求完整度 (16分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **范围事项清单** | 2.5 | 给出范围内事项清单，注明章节、版本号、页面引用，覆盖所有模块。 |
| **端到端流程图** | 2.5 | 输出端到端流程图，注明流程图所在章节/附件 + 每个节点的输入/输出。 |
| **主要场景覆盖** | 2.5 | 列出主要使用场景，包括触发条件、参与角色、成功判定。 |
| **异常流程标注** | 2.5 | 列出异常流程，标注触发条件、兜底策略、责任人。 |
| **状态流转规则** | 2 | 说明状态流转规则，包含状态枚举、进入/退出条件、互斥关系。 |
| **权限控制规则** | 2 | 标注权限控制规则，明确角色/租户/权限矩阵。 |
| **地区/语言差异** | 1 | 标注地区或语言差异，说明地区覆盖、开关策略、默认值。 |
| **原型/界面示意** | 1 | 提供关键原型或界面示意，标注版本、链接、更新时间。 |
| **异常/边界场景** | 1 | 异常/边界场景标注，需列出输入边界、性能上限、同步/异步要求。 |
| **需求无歧义性** | 1 | 需求描述无歧义，若存在 TBD 必须注明落地时间或责任人。 |

#### 2.1.2 可执行性 (8分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **版本优先级拆解** | 4 | 拆解 MLP 或版本优先级，需列出各里程碑覆盖范围、退出标准。 |
| **RACI/泳道图清晰度** | 4 | 涉及团队的 RACI/泳道图清晰，职责边界得到确认，需标注系统/角色/责任人/接口人。 |

#### 2.1.3 一致性 (6分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **术语定义统一** | 2 | 统一术语定义，附术语表位置或链接。 |
| **业务规则统一** | 2 | 统一业务规则描述，若多处引用规则，需注明统一出处。 |
| **数据口径统一** | 2 | 统一数据口径，包含指标公式、统计口径、刷新频率。 |

#### 2.1.4 体验与交互 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **关键路径原型** | 0.5 | 提供关键路径原型，标注版本、链接、更新时间、差异说明。 |
| **信息架构图** | 0.5 | 提供信息架构图，包含导航层级、入口、跨页面跳转。 |
| **组件规范对齐** | 0.5 | 对齐组件规范或设计系统，注明引用的组件库版本。 |
| **终端UI差异** | 0.5 | 说明不同终端的 UI 差异，列出适配范围、响应式策略。 |
| **响应时间目标** | 1 | 设定响应时间目标，例如首屏 ≤ Xs、交互反馈 ≤ Ys。 |
| **加载性能指标** | 1 | 设定加载性能指标，包括骨架屏/占位策略、最大加载时长。 |
| **空状态处理** | 0.5 | 描述空状态处理，列出触发条件、文案、行动引导。 |
| **错误提示策略** | 0.5 | 描述错误提示策略，需包含错误码、前端/后端兜底、可重试。 |

#### 2.1.5 用户价值 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **目标用户画像** | 1.5 | 明确目标用户画像，包含角色、规模、地区。 |
| **核心痛点/机会** | 1.5 | 列出核心痛点或机会，并列出影响指标/业务损失。 |
| **成功判定口径** | 1 | 定义成功判定口径，写明可观测指标、验收窗口。 |
| **风险影响评估** | 0.5 | 给出每项风险的影响评估，描述触发条件、量化影响。 |
| **缓解方案** | 0.5 | 提供对应的缓解方案，说明行动人、触发时机。 |

### 2.2 后端视角 (15分)

#### 2.2.1 业务范围 (4.5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **业务场景完整性** | 1 | 列出后端参与的全部业务场景，注明与现网流程的差异。 |
| **输入条件/鉴权** | 1 | 每个场景的输入条件、触发来源、所需鉴权或权限前置被描述清楚。 |
| **期望输出/副作用** | 1 | 每个场景的期望输出、副作用，写明落库/推送/回调及数据责任系统。 |
| **异常/失败路径** | 1 | 异常或失败路径逐条列出，标注是否需要后端兜底或补偿策略。 |
| **系统职责边界** | 0.5 | 系统职责边界清晰可见，注明读写矩阵、幂等键。 |

#### 2.2.2 技术方案准备度 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **端到端流程/时序图** | 1.5 | 提供端到端流程、关键时序、状态机或泳道图，便于推演系统交互。 |
| **跨系统输入输出约束** | 1 | 标注跨系统输入输出约束（格式、频率、时效），说明接口/字段示例。 |
| **容量/成本/合规假设** | 0.5 | 对容量、成本、合规等重要假设写明来源或数据。 |
| **技术风险描述** | 1 | 需求描述实现不存在技术风险，如有待定技术点必须描述风险与验证计划。 |

#### 2.2.3 系统依赖 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **内部系统/服务协同** | 0.5 | 枚举需协同的内部系统/服务，并附接口或联系人信息。 |
| **第三方/外部依赖** | 0.5 | 枚举第三方或外部依赖，说明合同/SLA、商务状态及预估lead time。 |
| **SLA/配额/限流要求** | 1 | 每个依赖的SLA、配额、限流、可用性要求在PRD中可查。 |
| **调用方式/时效** | 1 | 调用方式（同步/异步）、链路时效或批次节奏有描述。 |

#### 2.2.4 测试与验收 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **核心用例清单** | 0.5 | 给出后端必须覆盖的核心、异常、回归用例清单。 |
| **验收标准** | 0.5 | 验收标准写明成功/失败判定、日志或指标验证口径。 |
| **数据回填/回滚要求** | 1 | 数据回填、回滚、修复的业务要求提前声明。 |

#### 2.2.5 性能与容量 (1.5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **QPS/吞吐假设** | 0.5 | 峰值/平均QPS、吞吐、批处理窗口等业务假设写入PRD。 |
| **请求体量/延迟SLA** | 0.5 | 单次请求或消息的体量限制、延迟SLA在PRD中给出。 |
| **并发/限速/缓存约束** | 0.5 | 重点场景的并发、限速、缓存等约束或期望在PRD中体现。 |

---

### 2.3 前端视角 (15分)

#### 2.3.1 交互范围 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **场景/页面/入口清单** | 1 | 列出前端参与的全部场景/页面/入口，标注与现有产品的差异。 |
| **用户目标/触发路径** | 1 | 每个场景的用户目标、触发路径、角色权限被清晰描述。 |
| **范围外/延后交付** | 1 | 范围外/延后交付的交互被明确，避免后续scope膨胀。 |
| **协同责任边界** | 1 | 与服务端/其他端的协同责任边界被写明。 |

#### 2.3.2 设计交付 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **原型/视觉稿/组件规范** | 1 | 附带最新的原型、视觉稿、组件规范链接，说明历史版本差异。 |
| **关键交互描述** | 1 | 关键交互在PRD中有文字/动效描述，写明触发条件、边界、性能假设。 |
| **UI资产需求** | 1 | 列出所需的UI资产（图标、插画、视频），注明来源、交付格式。 |

#### 2.3.3 数据与接口 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **数据字段/来源/刷新** | 1 | 每个界面需要展示/编辑的数据字段、来源、刷新方式在PRD中说明。 |
| **缓存/本地存储** | 1 | 前端需要缓存/本地存储/离线的内容被标注，并交代失效策略。 |
| **错误码/占位数据** | 1 | 依赖后端返回的错误码、占位数据、分页策略等信息已给出或注明待补。 |

#### 2.3.4 状态与异常 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **所有状态覆盖** | 1 | 覆盖所有状态：加载、空、部分成功、失败、权限受限、超时等。 |
| **文案/按钮行为** | 1 | 每种状态的文案、按钮行为、可重试策略写在PRD或链接文档中。 |
| **表单校验规则** | 0.5 | 表单校验规则、前端兜底提示与服务端校验边界明确。 |
| **骨架屏/进度反馈** | 0.5 | 需要的骨架屏、进度反馈、批量操作反馈被描述。 |

#### 2.3.5 性能与体验 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **关键体验指标** | 0.5 | 提出关键体验指标：首屏时间、交互延时、动画帧率等目标或上限。 |
| **数据量/列表长度** | 0.5 | 数据量、列表长度、媒体大小等假设被描述。 |
| **预加载/懒加载** | 0.5 | 是否需要预加载、懒加载、占位策略等体验要求在PRD中明确。 |
| **弱网/低性能降级** | 0.5 | 对弱网/低性能设备的降级策略有业务层说明。 |

### 2.4 设计视角 (15分)

#### 2.4.1 业务与用户洞察 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **目标用户/场景/痛点** | 2 | 说明该版本的目标用户、核心场景、痛点及诉求数据来源。 |

#### 2.4.2 体验范围与流程 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **页面/模块/入口清单** | 2 | 列出涉及的页面/模块/入口/端到端流程，标注新增、改造或复用关系。 |
| **关键任务流程/状态流** | 2 | 关键任务流程、状态流、场景地图在PRD中可追溯，便于设计梳理信息架构。 |
| **跨角色/跨端交互边界** | 1 | 跨角色、跨终端、跨系统交互的边界与责任说明清楚。 |

#### 2.4.3 交互特性 (5分)

| 考核项 | 分值 | 评分标准 |

| **状态/空态/错误场景** | 2 | 状态、空态、加载、错误、权限受限等场景的业务期望被列举。 |
| **动效/反馈/提示要求** | 1 | 动效、反馈、提示的业务要求被写明，注明触发条件、持续时间。 |

### 2.5 测试视角 (15分)

| :--- | :--- | :--- |

| 考核项 | 分值 | 评分标准 |

| **错误提示/兜底策略** | 1.5 | 错误提示、兜底策略、告警机制写明，方便验证体验与告警链路。 |
| **灰度/回滚触发条件** | 1 | 灰度/回滚触发条件、限制条件记录，以备编写相应用例。 |

#### 2.5.4 性能与容量 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **峰值负载/并发** | 0.5 | 峰值负载、并发、批量处理窗口、消息体量等指标被描述。 |
| **性能/稳定性风险** | 0.5 | 性能/稳定性风险场景注明，QA可提前设计专项测试。 |
| **压测/基准测试** | 1 | 需开展压测、基准测试或混沌测试的范围、目标、数据准备方式在PRD中说明。 |

## 3. 一票否决项

如果出现以下情况，评审直接不通过：

1. **缺少范围事项清单或清单不完整。**
2. **缺少端到端流程图或流程图不清晰。**
3. **需求描述存在严重歧义或冲突。**
4. **缺少核心业务场景的验收标准。**
5. **涉及资金/权限/隐私的功能缺少安全红线定义。**
6. **缺少后端参与的业务场景列表或场景描述不完整。**
7. **缺少端到端流程/时序图或流程图不清晰。**
8. **系统依赖未明确或第三方依赖未说明。**
9. **缺少核心用例清单或验收标准。**
10. **缺少目标用户/场景/痛点说明。**
11. **缺少页面/模块/入口清单。**
12. **缺少关键任务流程/状态流。**
13. **缺少状态/空态/错误场景说明。**
14. **缺少前端参与的场景/页面/入口清单。**
15. **缺少原型/视觉稿/组件规范链接。**
16. **缺少数据字段/来源/刷新说明。**
17. **缺少状态覆盖或异常处理说明。**
18. **缺少业务流程/异常分支列表。**
19. **缺少验收标准或验收标准不明确。**
20. **缺少状态转移/重试/补偿流程说明。**
21. **缺少错误提示/兜底策略说明。**

## 4. 交付物核对清单

### 4.1 产品视角交付物

- [ ] 范围事项清单（注明章节、版本号、页面引用）
- [ ] 端到端流程图（注明流程图所在章节/附件 + 每个节点的输入/输出）
- [ ] 主要场景列表（包括触发条件、参与角色、成功判定）
- [ ] 异常流程列表（标注触发条件、兜底策略、责任人）
- [ ] 状态流转规则（包含状态枚举、进入/退出条件、互斥关系）
- [ ] 权限控制规则（明确角色/租户/权限矩阵）
- [ ] 版本优先级拆解（各里程碑覆盖范围、退出标准）
- [ ] RACI/泳道图（标注系统/角色/责任人/接口人）
- [ ] 术语表（术语定义统一）

## 5. 评分说明

### 5.1 评分原则

1. **证据导向**：每项打分严禁凭空判定，必须在PRD中找到对应依据。
2. **具体引用**：
   - **必须引用**：具体章节号、页码、表格标题或附件名称。
   - **必须摘录**：复制原文关键句或对逻辑进行转述（Paraphrase）。
3. **问题闭环**：对于扣分项，需明确指出缺失内容或逻辑冲突点，并提出具体补充期望。
4. **冲突处理**：若发现跨章节冲突，需在记录中明确两个出处并说明影响。

### 5.2 评分等级

- **优秀（90-100分）**：文档完整、清晰、可执行，各视角需求充分满足。
- **良好（80-89分）**：文档基本完整，存在少量非关键问题，不影响开发。
- **合格（60-79分）**：文档存在部分缺失或模糊，需要补充后才能进入开发。
- **不合格（<60分）**：文档存在严重缺失或一票否决项，需重新编写。
