# PRD Frontend Checklist

> 用于前端研发评审 PRD：确认业务描述、设计与依赖信息是否足以支撑前端技术方案与排期，确保无需在 PRD 中先写代码或实现细节。

## 交互范围（P0）
- [ ] PRD 列出前端参与的全部场景/页面/入口，并标注与现有产品的差异或复用关系。
- [ ] 每个场景的用户目标、触发路径、角色权限被清晰描述。
- [ ] 范围外/延后交付的交互被明确，避免后续 scope 膨胀。
- [ ] 与服务端/其他端的协同责任边界被写明（由谁渲染、由谁校验等）。

## 设计交付（P0）
- [ ] PRD 附带最新的原型、视觉稿、组件规范链接，并说明历史版本差异。
- [ ] 关键交互（拖拽、批量、快捷键等）在 PRD 中有文字/动效描述。
- [ ] 列出所需的 UI 资产（图标、插画、视频）。

## 数据与接口（P0）
- [ ] 每个界面需要展示/编辑的数据字段、来源、刷新方式在 PRD 中说明。
- [ ] 前端需要缓存/本地存储/离线的内容被标注，并交代失效策略。
- [ ] 依赖后端返回的错误码、占位数据、分页策略等信息已给出或注明待补。

## 状态与异常（P0）
- [ ] PRD 覆盖所有状态：加载、空、部分成功、失败、权限受限、超时等。
- [ ] 每种状态的文案、按钮行为、可重试策略写在 PRD 或链接文档中。
- [ ] 表单校验规则、前端兜底提示与服务端校验边界明确。
- [ ] 需要的骨架屏、进度反馈、批量操作反馈被描述。

## 终端与适配（P1）
- [ ] 指定需支持的浏览器、版本、内核、操作系统以及最低设备规格。
- [ ] 是否需响应式、自适应或多端（Web/H5/小程序/客户端）实现在 PRD 中说明。
- [ ] 第三方容器（WebView、嵌入式模块）限制或埋点接入要求被写明。

## 性能与体验（P1）
- [ ] PRD 提出关键体验指标：首屏时间、交互延时、动画帧率等目标或上限。
- [ ] 数据量、列表长度、媒体大小等假设被描述，便于评估虚拟列表/分片加载。
- [ ] 是否需要预加载、懒加载、占位策略等体验要求在 PRD 中明确。
- [ ] 对弱网/低性能设备的降级策略有业务层说明。

## 可访问性与本地化（P1）
- [ ] 多语言/多地区/多货币需求及默认语言在 PRD 中列出。
- [ ] 文案长度、日期/数字格式、RTL 布局等特殊要求被说明。
- [ ] 辅助提示（Tooltip、Help 文案、空状态说明）是否需要本地化被标注。

## 监控、埋点与运营（P2）
- [ ] 需埋点的事件、属性、触发条件在 PRD 或 Tracking Plan 中链接明确。
- [ ] 关键 UI/交互要暴露的监控指标、告警阈值写在 PRD。
