# PRD Checklist

> 说明：PRD 只需描述业务诉求与场景，不需要落具体 API 协议或库表结构，研发会基于 PRD 输出技术方案。

## 需求完整度（P0）
- [ ] 给出范围内事项清单。
- [ ] 输出端到端流程图。
- [ ] 列出主要使用场景。
- [ ] 列出异常流程。
- [ ] 说明状态流转规则。
- [ ] 标注权限控制规则。
- [ ] 标注地区或语言差异。
- [ ] 提供关键原型或界面示意。
- [ ] 异常/边界场景标注。
- [ ] 需求描述无歧义。
- [ ] 需求描述、流程无冲突。

## 可执行性（P1）
- [ ] 拆解 MLP 或版本优先级。
- [ ] 涉及团队的 RACI/泳道图清晰，职责边界得到确认。

## 一致性（P1）
- [ ] 统一术语定义。
- [ ] 统一业务规则描述。
- [ ] 统一数据口径。

## 体验与交互（P1）
- [ ] 提供关键路径原型。
- [ ] 提供信息架构图。
- [ ] 对齐组件规范或设计系统。
- [ ] 说明不同终端的 UI 差异。
- [ ] 设定响应时间目标。
- [ ] 设定加载性能指标。
- [ ] 描述空状态处理。
- [ ] 描述错误提示策略。
- [ ] 记录可用性要求。

## 技术可行性（P1）
- [ ] 描述跨系统/端的业务交互输入输出与约束（无需落字段级接口协议）。
- [ ] 描述关键数据要素的业务含义、来源与去向（无需设计库表结构）。
- [ ] 提供容量或并发假设。
- [ ] 列出兼容性矩阵（浏览器、OS、设备）。
- [ ] 标注性能约束。

## 用户价值（P2）
- [ ] PRD 概览提供一句话愿景、ROI/投入回报测算及优先级依据，方便决策者判断是否立项或推进。
- [ ] 业务指标或北极星调整的对齐记录/链接已附在 PRD 中，便于追溯。
- [ ] 明确目标用户画像。
- [ ] 量化目标用户规模。
- [ ] Persona、关键场景、痛点引用的调研或数据材料有明确出处（链接/附件编号）。
- [ ] 列出核心痛点或机会。
- [ ] 量化痛点的业务影响。
- [ ] 记录当前指标基线。
- [ ] 定义成功判定口径。

## 外部与历史依赖（P2）
- [ ] 列出第三方依赖（如有）。
- [ ] 关键依赖（法务/采购/第三方等）的 lead time 估算写入里程碑或风险提示（如有）。
- [ ] 标注涉及的历史需求链接（如果是迭代需求，需要有）。

## 数据与指标（P2）
- [ ] 列出核心业务指标。
- [ ] 写出指标计算公式。
- [ ] 定义每个埋点事件。
- [ ] 定义埋点字段及取值。
- [ ] 说明报表或看板需求。
- [ ] 定义数据访问权限。
- [ ] 设定上线监控项。
- [ ] 设定告警阈值。
- [ ] 描述数据验证或抽样方案。

## 风险与假设（P2）
- [ ] 给出每项风险的影响评估。
- [ ] 提供对应的缓解方案。
- [ ] 说明运营或通知计划。

## 上线前检查（P2）
- [ ] 明确验收标准，并在文档中给出可追溯的验收口径。
- [ ] 指定验收负责人并记录执行窗口/质检时间。
- [ ] 规划上线流程，并准备上线 Checklist、培训材料、客服 FAQ/脚本等资产。
- [ ] 规划灰度策略（含最近一次评审结论/版本记录）。
