## 1. 评分框架总览

本标准采用**五维综合评分机制**，总分100分，从产品、后端、前端、设计、测试五个专业视角全方位评估 PRD 质量。

| 维度 | 权重 | 核心考核点 |
| :--- | :--- | :--- |
| **产品视角** | **40%** | 决策卡完整性、可交互Demo、可执行规格、上下文治理、格式规范 |
| **后端视角** | **15%** | 业务范围、技术方案准备度、系统依赖、测试验收、性能容量 |
| **前端视角** | **15%** | 交互范围、设计交付、数据接口、状态异常、性能体验 |
| **设计视角** | **15%** | 业务洞察、体验范围、交互特性 |
| **测试视角** | **15%** | 场景覆盖、判定口径、状态异常、性能容量 |

## 2. 评分细则

### 2.1 产品视角 (40分)

#### 2.1.1 决策卡 (14分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **用户与场景** | 3 | 明确定义目标用户群体及具体使用场景，包含用户画像、时间地点环境描述。 |
| **痛点定义** | 3 | 痛点描述清晰且可度量（如"耗时30分钟"而非"耗时很长"），提供数据支撑。 |
| **关键假设** | 3 | 列出关键假设且假设可证伪，区分已知事实和待验证假设。 |
| **成功指标** | 4 | 包含至少2个过程指标和1个结果指标，明确计算公式、基准值和目标值。 |
| **红线/不做什么** | 1 | 明确合规红线及Non-goals（本次不做的范围）。 |

#### 2.1.2 可交互Demo (14分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **可运行性** | 4 | 必须是能点能跑的版本（HTML/React等），主路径可完整跑通。 |
| **异常/边界覆盖** | 4 | 覆盖至少2-3个关键异常或边界状态（空数据、加载失败、权限不足等）。 |
| **Mock数据质量** | 3 | 使用真实感Mock数据，包含边界情况，严禁全是"test/123"。 |
| **交互完整性** | 2 | 包含关键交互细节，如删除/敏感操作的二次确认、状态反馈。 |
| **设计规范遵循** | 1 | Prompt中明确使用公司统一设计系统/组件库。 |

#### 2.1.3 可执行规格 (6分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **可测试性格式** | 4 | 采用Given/When/Then格式或结构化清单编写，直接对应测试用例。 |
| **验收标准明确** | 2 | Criteria清晰无歧义，包含无权限返回、驳回表现、防重机制等。 |

#### 2.1.4 上下文治理 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **规则入库** | 3 | 核心业务规则、安全要求、术语表等沉淀为独立文件。 |
| **事实库完整** | 1 | 包含目标、术语、约束、规则、示例数据、埋点要求等。 |

#### 2.1.5 格式规范 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **Markdown编写** | 1 | 文档主体使用Markdown编写，便于AI读取和版本管理。 |
| **图表代码化** | 1 | 流程图、时序图使用Mermaid代码输出，严禁使用图片。 |

### 2.2 后端视角 (15分)

#### 2.2.1 业务范围 (4.5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **业务场景完整性** | 1 | 列出后端参与的全部业务场景，注明与现网流程的差异。 |
| **输入条件/鉴权** | 1 | 每个场景的输入条件、触发来源、所需鉴权或权限前置被描述清楚。 |
| **期望输出/副作用** | 1 | 每个场景的期望输出、副作用，写明落库/推送/回调及数据责任系统。 |
| **异常/失败路径** | 1 | 异常或失败路径逐条列出，标注是否需要后端兜底或补偿策略。 |
| **系统职责边界** | 0.5 | 系统职责边界清晰可见，注明读写矩阵、幂等键。 |

#### 2.2.2 技术方案准备度 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **端到端流程/时序图** | 1.5 | 提供端到端流程、关键时序、状态机或泳道图，便于推演系统交互。 |
| **跨系统输入输出约束** | 1 | 标注跨系统输入输出约束（格式、频率、时效），说明接口/字段示例。 |
| **容量/成本/合规假设** | 0.5 | 对容量、成本、合规等重要假设写明来源或数据。 |
| **技术风险描述** | 1 | 需求描述实现不存在技术风险，如有待定技术点必须描述风险与验证计划。 |

#### 2.2.3 系统依赖 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **内部系统/服务协同** | 0.5 | 枚举需协同的内部系统/服务，并附接口或联系人信息。 |
| **第三方/外部依赖** | 0.5 | 枚举第三方或外部依赖，说明合同/SLA、商务状态及预估lead time。 |
| **SLA/配额/限流要求** | 1 | 每个依赖的SLA、配额、限流、可用性要求在PRD中可查。 |
| **调用方式/时效** | 1 | 调用方式（同步/异步）、链路时效或批次节奏有描述。 |

#### 2.2.4 测试与验收 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **核心用例清单** | 0.5 | 给出后端必须覆盖的核心、异常、回归用例清单。 |
| **验收标准** | 0.5 | 验收标准写明成功/失败判定、日志或指标验证口径。 |
| **数据回填/回滚要求** | 1 | 数据回填、回滚、修复的业务要求提前声明。 |

#### 2.2.5 性能与容量 (1.5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **QPS/吞吐假设** | 0.5 | 峰值/平均QPS、吞吐、批处理窗口等业务假设写入PRD。 |
| **请求体量/延迟SLA** | 0.5 | 单次请求或消息的体量限制、延迟SLA在PRD中给出。 |
| **并发/限速/缓存约束** | 0.5 | 重点场景的并发、限速、缓存等约束或期望在PRD中体现。 |

### 2.3 前端视角 (15分)

#### 2.3.1 交互范围 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **场景/页面/入口清单** | 1 | 列出前端参与的全部场景/页面/入口，标注与现有产品的差异。 |
| **用户目标/触发路径** | 1 | 每个场景的用户目标、触发路径、角色权限被清晰描述。 |
| **范围外/延后交付** | 1 | 范围外/延后交付的交互被明确，避免后续scope膨胀。 |
| **协同责任边界** | 1 | 与服务端/其他端的协同责任边界被写明。 |

#### 2.3.2 设计交付 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **原型/视觉稿/组件规范** | 1 | 附带最新的原型、视觉稿、组件规范链接，说明历史版本差异。 |
| **关键交互描述** | 1 | 关键交互在PRD中有文字/动效描述，写明触发条件、边界、性能假设。 |
| **UI资产需求** | 1 | 列出所需的UI资产（图标、插画、视频），注明来源、交付格式。 |

#### 2.3.3 数据与接口 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **数据字段/来源/刷新** | 1 | 每个界面需要展示/编辑的数据字段、来源、刷新方式在PRD中说明。 |
| **缓存/本地存储** | 1 | 前端需要缓存/本地存储/离线的内容被标注，并交代失效策略。 |
| **错误码/占位数据** | 1 | 依赖后端返回的错误码、占位数据、分页策略等信息已给出或注明待补。 |

#### 2.3.4 状态与异常 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **所有状态覆盖** | 1 | 覆盖所有状态：加载、空、部分成功、失败、权限受限、超时等。 |
| **文案/按钮行为** | 1 | 每种状态的文案、按钮行为、可重试策略写在PRD或链接文档中。 |
| **表单校验规则** | 0.5 | 表单校验规则、前端兜底提示与服务端校验边界明确。 |
| **骨架屏/进度反馈** | 0.5 | 需要的骨架屏、进度反馈、批量操作反馈被描述。 |

#### 2.3.5 性能与体验 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **关键体验指标** | 0.5 | 提出关键体验指标：首屏时间、交互延时、动画帧率等目标或上限。 |
| **数据量/列表长度** | 0.5 | 数据量、列表长度、媒体大小等假设被描述。 |
| **预加载/懒加载** | 0.5 | 是否需要预加载、懒加载、占位策略等体验要求在PRD中明确。 |
| **弱网/低性能降级** | 0.5 | 对弱网/低性能设备的降级策略有业务层说明。 |

### 2.4 设计视角 (15分)

#### 2.4.1 业务与用户洞察 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **目标用户/场景/痛点** | 2 | 说明该版本的目标用户、核心场景、痛点及诉求数据来源。 |
| **成功指标/体验目标** | 2 | 成功指标、体验目标明确，可指导设计评估，注明指标口径、时间窗口。 |
| **业务限制/假设** | 1 | 业务限制或假设对体验的约束被列出，写明不可突破的限制与可调整范围。 |

#### 2.4.2 体验范围与流程 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **页面/模块/入口清单** | 2 | 列出涉及的页面/模块/入口/端到端流程，标注新增、改造或复用关系。 |
| **关键任务流程/状态流** | 2 | 关键任务流程、状态流、场景地图在PRD中可追溯，便于设计梳理信息架构。 |
| **跨角色/跨端交互边界** | 1 | 跨角色、跨终端、跨系统交互的边界与责任说明清楚。 |

#### 2.4.3 交互特性 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **特殊交互说明** | 2 | 特殊交互在PRD中说明动线与约束，写明输入上限、延迟容忍度。 |
| **状态/空态/错误场景** | 2 | 状态、空态、加载、错误、权限受限等场景的业务期望被列举。 |
| **动效/反馈/提示要求** | 1 | 动效、反馈、提示的业务要求被写明，注明触发条件、持续时间。 |

### 2.5 测试视角 (15分)

#### 2.5.1 场景覆盖 (5分)

#### 2.5.2 需求判定口径 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **验收标准** | 2 | 每个需求具备可量化或可验证的验收标准，列出阈值、统计窗口、取数方式。 |

#### 2.5.3 状态与异常 (4分)

| 考核项 | 分值 | 评分标准 |

| **灰度/回滚触发条件** | 1 | 灰度/回滚触发条件、限制条件记录，以备编写相应用例。 |

#### 2.5.4 性能与容量 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **峰值负载/并发** | 0.5 | 峰值负载、并发、批量处理窗口、消息体量等指标被描述。 |
| **性能/稳定性风险** | 0.5 | 性能/稳定性风险场景注明，QA可提前设计专项测试。 |
| **压测/基准测试** | 1 | 需开展压测、基准测试或混沌测试的范围、目标、数据准备方式在PRD中说明。 |

## 3. 一票否决项

如果出现以下情况，评审直接不通过：

1. **只有PRD文档，没有可交互Demo。**
2. **Demo仅为静态图片拼接，无法进行实际输入/点击交互。**
3. **关键数据表单未定义验证规则（如长度、类型、必填）。**
4. **涉及资金/权限/隐私的功能缺少安全红线定义。**
5. **缺少后端参与的业务场景列表或场景描述不完整。**
6. **缺少端到端流程/时序图或流程图不清晰。**
7. **系统依赖未明确或第三方依赖未说明。**
8. **缺少核心用例清单或验收标准。**
9. **缺少目标用户/场景/痛点说明。**
10. **缺少页面/模块/入口清单。**
11. **缺少关键任务流程/状态流。**
12. **缺少状态/空态/错误场景说明。**
13. **缺少前端参与的场景/页面/入口清单。**
14. **缺少原型/视觉稿/组件规范链接。**
15. **缺少数据字段/来源/刷新说明。**
16. **缺少状态覆盖或异常处理说明。**
17. **缺少业务流程/异常分支列表。**
18. **缺少验收标准或验收标准不明确。**
19. **缺少状态转移/重试/补偿流程说明。**
20. **缺少错误提示/兜底策略说明。**

## 4. 交付物核对清单

### 4.1 产品视角交付物

- [ ] 一页决策卡（包含指标与红线）
- [ ] 可交互Demo（主路径+异常态+真实数据）
- [ ] 可执行规格（Given/When/Then）
- [ ] 上下文文件（规则、术语、约束入库）

### 4.2 后端视角交付物

- [ ] 业务场景列表（触发入口、业务前置条件、是否新增/改造）
- [ ] 输入条件/鉴权说明（鉴权方式、租户限制、失败提示）
- [ ] 期望输出/副作用说明（落库/推送/回调及数据责任系统）
- [ ] 异常/失败路径列表（补偿链路/重试次数）
- [ ] 系统职责边界说明（读写矩阵、幂等键）
- [ ] 端到端流程/时序图（泳道参与方、时序编号）
- [ ] 跨系统输入输出约束（接口/字段示例、同步与否）
- [ ] 容量/成本/合规假设（测算时间、数据来源、假设条件）
- [ ] 系统依赖清单（内部系统/服务协同、第三方/外部依赖）
- [ ] SLA/配额/限流要求（当前SLA vs 需求SLA的差异）
- [ ] 核心用例清单（用例编号、触发数据、期望日志）
- [ ] 验收标准（验证工具、监控来源）

### 4.3 前端视角交付物

- [ ] 场景/页面/入口清单（导航路径、入口截图/链接）
- [ ] 用户目标/触发路径说明（权限校验位置、失败提示）
- [ ] 范围外/延后交付列表（排期/版本计划）
- [ ] 协同责任边界说明（接口责任方、同步方式、缓存策略）
- [ ] 原型/视觉稿/组件规范（版本号、更新时间、记录位置）
- [ ] 关键交互描述（触发条件、边界、性能假设）
- [ ] UI资产清单（来源、交付格式、体积限制）
- [ ] 数据字段/来源/刷新说明（字段枚举、默认值、是否必填）
- [ ] 缓存/本地存储说明（存储键、刷新频率、冲突处理）
- [ ] 状态覆盖列表（状态触发条件与兜底策略）
- [ ] 表单校验规则（同步/异步校验点、错误码对齐）
- [ ] 关键体验指标（首屏时间、交互延时、动画帧率）

### 4.4 设计视角交付物

## 5. 评分说明

### 5.1 评分原则

1. **证据导向**：每项打分严禁凭空判定，必须在PRD中找到对应依据。
2. **具体引用**：
   - **必须引用**：具体章节号、页码、表格标题或附件名称。
   - **必须摘录**：复制原文关键句或对逻辑进行转述（Paraphrase）。
3. **问题闭环**：对于扣分项，需明确指出缺失内容或逻辑冲突点，并提出具体补充期望。
4. **冲突处理**：若发现跨章节冲突，需在记录中明确两个出处并说明影响。

### 5.2 评分等级

- **优秀（90-100分）**：文档完整、清晰、可执行，各视角需求充分满足。
- **良好（80-89分）**：文档基本完整，存在少量非关键问题，不影响开发。
- **合格（60-79分）**：文档存在部分缺失或模糊，需要补充后才能进入开发。
- **不合格（<60分）**：文档存在严重缺失或一票否决项，需重新编写。
