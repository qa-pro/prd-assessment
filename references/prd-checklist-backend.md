# PRD Backend Checklist

> 说明：本清单供后端研发评审 PRD 时使用，用于判断 PRD 是否提供了产出技术方案所需的业务上下文、依赖与约束。若某项缺失，应及时驱动产品补齐 PRD，而非在 PRD 中直接写技术方案细节。

## 业务范围（P0）
- [ ] PRD 列出后端参与的全部业务场景，并注明与现网流程的差异。
- [ ] 每个场景的输入条件、触发来源、所需鉴权或权限前置被描述清楚。
- [ ] 每个场景的期望输出、副作用。
- [ ] 异常或失败路径逐条列出，标注是否需要后端兜底或补偿策略。
- [ ] 系统职责边界（数据主责任系统、写入/消费方、幂等要求）在 PRD 中清晰可见。

## 技术方案准备度（P0）
- [ ] PRD 提供端到端流程、关键时序、状态机或泳道图，便于推演系统交互。
- [ ] PRD 标注跨系统输入输出约束（格式、频率、时效），说明哪些细节留待技术方案定义。
- [ ] 对容量、成本、合规等重要假设，PRD 写明来源或数据，便于技术方案引用。
- [ ] 需求描述实现不存在技术风险。

## 系统依赖（P0）
- [ ] PRD 枚举需协同的内部系统/服务，并附接口或联系人信息。
- [ ] PRD 枚举第三方或外部依赖，说明合同/SLA、商务状态及预估 lead time，如果无第三方依赖，认为已完成。
- [ ] 每个依赖的 SLA、配额、限流、可用性要求在 PRD 中可查。
- [ ] 调用方式（同步/异步）、链路时效或批次节奏在 PRD 中有描述，若待定需注明。
- [ ] 认证/鉴权、密钥或凭证的责任方和交付方式被提前说明。

## 测试与验收（P0）
- [ ] PRD 给出后端必须覆盖的核心、异常、回归用例清单，并能映射到场景。
- [ ] 验收标准写明成功/失败判定、日志或指标验证口径。
- [ ] 数据回填、回滚、修复的业务要求在 PRD 中提前声明，包含触发条件。

## 性能与容量（P1）
- [ ] 峰值/平均 QPS、吞吐、批处理窗口等业务假设写入 PRD，并说明测算依据。
- [ ] 单次请求或消息的体量限制、延迟 SLA、后台作业 SLA 在 PRD 中给出。
- [ ] 重点场景的并发、限速、缓存等约束或期望在 PRD 中体现，方便技术方案取用。
- [ ] 若需压测或容量验证，PRD 标注目标指标与完成时间节点。

## 数据迁移与灰度（P1）
- [ ] PRD 描述需要迁移/双写的数据范围、体量、影响系统。
- [ ] 迁移/双写/回填的成功判定口径、样本量、时间窗口在 PRD 中说明。
- [ ] 迁移失败、灰度终止、回滚的触发条件与业务补救方式已定义。

## 部署与运维（P2）
- [ ] PRD 标注上线范围（地域/租户/环境）及灰度策略期望，便于后端规划发布方案。
- [ ] 对配置项、开关、客服/运营切换动作的业务要求被列出。
- [ ] 回滚或应急处理需要达到的业务结果、通知对象在 PRD 中说明。
- [ ] 对外沟通、客服、运营脚本等上线配套材料是否准备到位在 PRD 中有迹可循。

## 安全与合规（P2）
- [ ] PRD 罗列认证、授权、权限的业务要求以及与现有体系的差异。
- [ ] 需要保护的敏感字段、脱敏/加密策略、访问审计诉求在 PRD 中定义。
- [ ] 适用的监管/合规、数据主权/地域限制被写清。
- [ ] 数据留存周期、访问最小化等要求有据可依，方便技术方案落实。

## 观测与告警（P2）
- [ ] PRD 指定需要监控的业务指标，并说明上线判定或告警阈值。
- [ ] 需要监控的系统级指标（错误率、耗时、资源使用）在 PRD 中被提及。
- [ ] 故障通知路径、值班责任、响应 SLA 有清晰描述。
- [ ] 需要的日志、追踪、埋点或报表要求写入 PRD，避免技术方案臆测。
